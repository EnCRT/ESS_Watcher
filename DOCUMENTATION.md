# 📖 ESS Watcher — Полная техническая документация

Документ описывает аппаратную часть, программный стек, внутреннюю архитектуру прошивки, а также содержит полный справочник переменных и методов.

## 1. Принцип работы устройства

Устройство **постоянно** проверяет наличие напряжения 220В через цифровой вход. Источник «220В есть» определяется не прямым измерением сети, а по наличию стабилизированного питания 5В, приходящего от UPS-модуля, который запитан от сети через Type-C. Питание самой платы ESP32-C3 при этом идёт от аккумулятора 18650.

**Логика работы:**

```
┌─────────────┐    220В    ┌──────────────┐   5В (Type-C)  ┌──────────────┐
│  Розетка    │───────────▶│ UPS Module   │───────────────▶│ Делитель напр.│
│  220В       │            │  +18650 АКБ  │                │  5В → 3.3В    │
└─────────────┘            └──────────────┘                └──────┬───────┘
                                                                  │ GPIO 0 (Sense)
                                                                  ▼
                                                          ┌──────────────┐
                                                          │  ESP32-C3    │
                                                          │  SuperMini   │
                                                          └──────┬───────┘
                                                                 │ WiFi
                                                                 ▼
                                                          ┌──────────────┐
                                                          │ Telegram Bot │
                                                          └──────────────┘
```

1. **Сеть 220В есть** → UPS выдаёт 5В → на `GPIO 0` логическая единица → статус «питание есть».
2. **Сеть 220В пропала** → UPS переключается на АКБ, перестаёт выдавать 5В на sense-линию → на `GPIO 0` ноль → устройство фиксирует «отключение 220В».
3. Устройство отправляет сообщение в Telegram через Wi-Fi.
4. При восстановлении сети отправляется другое сообщение.
5. Аккумулятор 18650 обеспечивает автономность: сообщение об отключении **успевает** уйти в Telegram, даже если основное электричество уже пропало.

**Визуальная индикация (RGB-светодиод):** двойной «пых» раз в 2 секунды.

| Цвет | Состояние |
|------|-----------|
| 🟢 Зелёный | Сеть 220В есть |
| 🔴 Красный | Работа от АКБ (220В пропала) |
| 🔵 Синий | Режим настройки (Wi-Fi точка доступа) |

**Звуковая индикация:** проигрывание RTTTL-мелодий через пассивный зуммер KY-006 при смене состояния питания.

## 2. Аппаратная часть

### 2.1 Комплектующие

| № | Модуль | Назначение |
|---|--------|------------|
| 1 | **ESP32-C3 SuperMini** | Основной контроллер: RISC-V одноядерный процессор, 4 МБ Flash, Wi-Fi 2.4 ГГц + BLE 5. |
| 2 | **KY-006 Passive Buzzer** | Пассивный пьезозуммер. Воспроизводит мелодии (генерация тонов через ШИМ). |
| 3 | **UPS Module 5V/2A** | Модуль заряда/разряда Li-Ion с выходом 5В (Type-C). Обеспечивает бесперебойное питание и резерв от 18650. |
| 4 | **WS2812B RGB LED** | Адресный светодиод для визуальной индикации статуса. |
| 5 | **Аккумулятор 18650 (Samsung 35E)** | Буферный источник питания. Обеспечивает работу устройства при пропадании сети. |

### 2.2 Назначение пинов ESP32-C3

| Пин | Назначение | Комментарий |
|-----|------------|-------------|
| **GPIO 0** | `PIN_POWER_SENSE` — мониторинг 220В | **Обязательно** через резистивный делитель (5В → 3.3В, напр. 10кОм/20кОм). Инициализируется как `INPUT_PULLDOWN`. |
| **GPIO 2** | `PIN_BUZZER` — пассивный зуммер | Управляется аппаратным ШИМ через LEDC-канал 0. |
| **GPIO 9** | `PIN_CONFIG_BUTTON` — кнопка настройки | Совпадает с кнопкой BOOT на плате. `INPUT_PULLUP`, активный уровень — LOW. |
| **GPIO 10** | `PIN_LED_DATA` — WS2812B | Линия данных адресного светодиода (через библиотеку FastLED). |

> ⚠️ **Важно по GPIO 8:** встроенный светодиод платы (_strapping pin_ GPIO 8) намеренно **не используется**. Изменение его состояния при загрузке приводило к сбоям старта.

### 2.3 Энергоэффективность и терморежим

- **Мощность передатчика** принудительно ограничивается `WIFI_POWER_5dBm` при вызове `WiFi.begin()` — снижает пиковые токи потребления и нагрев чипа.
- **Режим сна модема** `WIFI_PS_MIN_MODEM` включается после получения IP-адреса — экономит заряд АКБ при работе в автономном режиме.
- Снижение мощности критично: на штатных 20 dBm чип нагревается до 60+ °C и выше.

- `board_build.filesystem = littlefs` — включение LittleFS.
- `extra_script.py` — автоматически запускает `pio run --target uploadfs` (заливка звуков) после каждой прошивки.

### 5.3 Справочник функций

#### `setRGBColor(CRGB color)`
Записывает цвет в `leds[0]` и вызывает `FastLED.show()`. Базовый примитив управления светодиодом.

#### `playMelody(const char* melodyFile, bool powerOn)`
Воспроизводит мелодию по имени файла из `/sounds/`.
- Читает файл из LittleFS, пропускает комментарии (`#`) и пустые строки, берёт первую непустую RTTTL-строку.
- Если файла нет/пустой — **fallback**: проигрывает встроенную короткую RTTTL (`StdOn` или `StdOff` в зависимости от `powerOn`).
- Параметр `powerOn` используется только для выбора fallback-мелодии.

#### `handleStatusIndicationAsync()`
Неблокирующая функция индикации статуса. Вызывается в каждом цикле `loop()`.
- Раз в `BLINK_INTERVAL` (2 сек) запускает цикл двойного «пыха».
- Выбирает цвет: **синий** в Config Mode, **зелёный** при наличии сети, **красный** в остальных случаях.
- Распределение вспышек по времени внутри цикла:
  - `0–60 мс` — горит;
  - `60–160 мс` — погашен;
  - `160–220 мс` — горит (вторая вспышка);
  - `>220 мс` — погашен до конца 2-секундного цикла.

#### `checkNTPStatusAsync()`
Проверяет, синхронизировалось ли время (раз в 1 секунду). Если `time(nullptr) > 24 * 3600` — значит NTP вернул осмысленное время (больше суток с 1970 года). После этого:
- устанавливает `isTimeSynced = true`;
- формирует флаг `needsStartupMessage = true`, чтобы в `loop()` ушло стартовое сообщение.

#### `sendTelegramMessage(String msg)`
Отправляет текстовое сообщение в Telegram (см. подробности в §9).
- Возвращает `true` только если HTTP-код ответа равен `200`.
- Блокируется, если Wi-Fi не подключён или время не синхронизировано.

#### `handleRoot()`
HTTP-обработчик `GET /`. Отдаёт HTML-страницу `WEB_PORTAL_HTML`, подставляя текущие значения настроек вместо плейсхолдеров вида `{{ssid}}`, `{{token}}` и т.д.

#### `handleSave()`
HTTP-обработчик `POST /save`. Принимает поля формы (`ssid`, `pass`, `token`, `chat`, `thread`, `melody_on`, `melody_off`, четыре текста сообщений, и т.д.), сохраняет их через `configManager.save()` и перезагружает устройство (`ESP.restart()`).

#### `handlePlay()`
HTTP-обработчик `/play?file=...`. Читает RTTTL-строку из файла `/sounds/<file>` и проигрывает её (используется кнопкой ▶ в веб-интерфейсе для предпрослушивания).

#### `handleVolume()`
HTTP-обработчик `/volume?vol=...`. Устанавливает громкость зуммера (0–100%), сохраняет в NVS.

#### `enterConfigMode()`
Переводит устройство в режим точки доступа:
- `WiFi.mode(WIFI_AP)` + `softAP("ESS Watcher", "12345678")` на IP `192.168.0.11`.
- Запускает DNS-сервер, перенаправляющий **все** запросы на `192.168.0.11` (Captive Portal).
- Регистрирует HTTP-обработчики и играет приветственный звук «wifi_mode».

#### `WiFiEvent(WiFiEvent_t event, WiFiEventInfo_t info)`
Событийный обработчик Wi-Fi (см. §7). Реагирует на два события:
- `ARDUINO_EVENT_WIFI_STA_GOT_IP` — Wi-Fi подключён, получен IP → взводит флаги, включает sleep-режим, запускает NTP.
- `ARDUINO_EVENT_WIFI_STA_DISCONNECTED` — сбрасывает `isWifiConnected`, логирует причину, инициирует переподключение.

### 5.4 Жизненный цикл: setup() и loop()

#### `setup()`
Выполняется один раз при старте:
1. `delay(2000)` — дать USB-CDC время инициализироваться в ОС.
2. `Serial.begin(115200)` — запуск отладочного вывода.
3. `configManager.begin()` — монтирование NVS, загрузка/инициализация настроек.
4. `SoundStorage::begin()` — монтирование LittleFS, `listSounds()` — вывод доступных мелодий в Serial.
5. Конфигурация пинов: `PIN_POWER_SENSE` → `INPUT_PULLDOWN`, `PIN_CONFIG_BUTTON` → `INPUT_PULLUP`.
6. `FastLED.addLeds<WS2812B, 10, GRB>(...)` — инициализация светодиода.
7. `buzzer.begin(2, 0)` + применение сохранённой громкости.
8. Установка корневого сертификата Telegram в `secured_client`.
9. Чтение **начального** состояния питания (без антидребезга — мы уже знаем текущее состояние при старте).
10. Проигрывание звука «startup».
11. Подключение обработчика `WiFi.onEvent(WiFiEvent)`.
12. `WiFi.setTxPower(WIFI_POWER_5dBm)` — ограничение мощности **до** `begin()`.
13. `WiFi.setAutoReconnect(true)` + `WiFi.begin(...)`.

#### `loop()`
Главный цикл. **Сначала** проверяет флаг `isConfigMode` — если активен, обрабатывает только DNS, веб-клиентов, зуммер и индикацию, затем сразу выходит из итерации (возврат в режим настройки не должен опрашивать питание).

В нормальном режиме:
1. **Heartbeat** — каждые 5 секунд выводит в Serial статус Wi-Fi и питания.
2. `checkNTPStatusAsync()` — проверка готовности NTP.
3. `buzzer.handleAsync()` — продвижение RTTTL-проигрывателя.
4. `handleStatusIndicationAsync()` — обновление светодиода.
5. **Обработка кнопки** — детекция удержания ≥ `CONFIG_HOLD_TIME` → `enterConfigMode()`.
6. **Стартовое сообщение** — если `needsStartupMessage`, отправляет соответствующее сообщение (сеть/АКБ).
7. **Мониторинг питания** (см. §6) — основная логика устройства.
8. `delay(10)` — минимальная разгрузка цикла.

## 6. Мониторинг сети 220В (алгоритм)

Лежит в `loop()`, блок 4. Алгоритм с **антидребезгом через накопление подтверждений**:

```
Каждые POWER_CHECK_INTERVAL (500 мс):
  currentReading = digitalRead(PIN_POWER_SENSE)

  если currentReading == potentialPowerState:
      debounceCounter++                       # подтверждаем кандидат
  иначе:
      debounceCounter = 0                      # сброс
      potentialPowerState = currentReading    # новый кандидат

  если debounceCounter >= DEBOUNCE_THRESHOLD (5):
      # Состояние стабильно держится 5×500мс = 2.5 сек
      если potentialPowerState != lastPowerState:
          lastPowerState = potentialPowerState
          isPowerOn = potentialPowerState

          если isPowerOn:
              playMelody(melody_on_file, true)
              sendTelegramMessage(msg_power_restored)
          иначе:
              playMelody(melody_off_file, false)
              sendTelegramMessage(msg_power_lost)
```

**Зачем антидребезг:** кратковременные просадки/помехи и переходные процессы на UPS не должны вызывать ложных уведомлений. Требуется 2.5 секунды **стабильного** нового состояния, прежде чем сработает оповещение.

**Три переменные состояния:**
- `potentialPowerState` — что мы сейчас «подозреваем» (последнее «сырое» чтение).
- `debounceCounter` — сколько раз подряд подтвердилось подозрение.
- `lastPowerState` — последнее **подтверждённое** состояние. Именно с ним сравнивается кандидат — если они разошлись, значит событие реальное.

## 7. WiFi и событийная модель

Подход **событийно-ориентированный**, а не «блокирующий опрос в `loop()`». Вся работа с Wi-Fi делегирована системному обработчику `WiFi.onEvent()`.

### Почему не блокирующий опрос?

При «классическом» подходе в `loop()` стоит `while (WiFi.status() != WL_CONNECTED) delay(500);` — это вешает цикл и мешает остальной логике (зуммеру, индикации, кнопке). Событийная модель позволяет `loop()` крутиться без задержек.

### Обрабатываемые события (`WiFiEvent`)

#### `ARDUINO_EVENT_WIFI_STA_GOT_IP`
Wi-Fi подключён, получен IP-адрес.
- `isWifiConnected = true`
- Логирование IP в Serial
- `WiFi.setSleep(WIFI_PS_MIN_MODEM)` — переход в экономичный режим модема
- Если NTP ещё не запущен → `configTime(0, 0, "pool.ntp.org", "time.nist.gov")` и `ntpSyncStarted = true`

#### `ARDUINO_EVENT_WIFI_STA_DISCONNECTED`
Соединение потеряно.
- `isWifiConnected = false`
- Логируется причина отключения (`info.wifi_sta_disconnected.reason`) — код ошибки из таблицы ESP-IDF
- Если **не** в Config Mode → `WiFi.reconnect()` (дополнительно работает `setAutoReconnect(true)`)

### Почему NTP-инициализация именно здесь?

Стек SSL/TLS на ESP32 (mbedTLS) проверяет срок действия сертификата `api.telegram.org`. Без синхронизации времени часы показывают ~1970 год, и любой сертификат отклоняется как «ещё не действительный». Поэтому:
- NTP запускается сразу после получения IP.
- Отправка в Telegram **запрещена**, пока `isTimeSynced == false` (см. `sendTelegramMessage`).

### Почему тяжёлая обработка вынесена в `loop()`, а не в обработчик события?

Задача `arduino_events` в стеке ESP-IDF имеет стек всего ~4 КБ. SSL-рукопожатие требует >6 КБ. Поэтому внутри `WiFiEvent()` только взводятся **флаги**, а вся реальная работа (NTP, SSL, POST) выполняется в `loop()` из основной задачи с большим стеком.

## 8. Режим настройки (Config Mode / Captive Portal)

### Вход в режим

Удержание кнопки BOOT (`GPIO 9`) ≥ 2 секунд (`CONFIG_HOLD_TIME`). Вызывается `enterConfigMode()`:

```cpp
WiFi.mode(WIFI_AP);
WiFi.softAPConfig(IPAddress(192,168,0,11), IPAddress(192,168,0,11), IPAddress(255,255,255,0));
WiFi.softAP("ESS Watcher", "12345678");

dnsServer.start(53, "*", IPAddress(192,168,0,11));  // перехват ВСЕХ DNS
```

### Captive Portal

`DNSServer` слушает UDP-порт 53 и на **любой** DNS-запрос отвечает `192.168.0.11`. Это заставляет ОС телефона/ПК показать всплывающее окно «Войти в сеть», которое сразу открывает страницу настройки. Также `server.onNotFound(handleRoot)` возвращает страницу настроек на любой неизвестный URL — на случай прямого ввода адреса.

| Параметр точки доступа | Значение |
|------------------------|----------|
| SSID | `ESS Watcher` |
| Пароль | `12345678` |
| IP шлюза/устройства | `192.168.0.11` |
| Маска | `255.255.255.0` |

### Маршруты веб-сервера

| Метод + путь | Обработчик | Назначение |
|--------------|------------|------------|
| `GET /` (и любой unknown) | `handleRoot` | Отдаёт HTML-форму с текущими настройками |
| `POST /save` | `handleSave` | Сохраняет настройки и перезагружает |
| `GET /play?file=...` | `handlePlay` | Предпрослушать выбранную мелодию |
| `GET /volume?vol=...` | `handleVolume` | Установить громкость зуммера |

### Поведение устройства в Config Mode

В `loop()` первой же проверкой `if (isConfigMode) { ... return; }` — устройство обслуживает только портал и не опрашивает питание. Цвет светодиода — синий.

## 9. Telegram: отправка уведомлений

### Почему НЕ используется `bot.sendMessage()` из `UniversalTelegramBot`

Две критические причины, зафиксированные в `implementation_plan.md`:

1. **Поддержка топиков супергрупп.** Библиотека v1.3.0 не умеет корректно передавать параметр `message_thread_id` (путает его с `reply_to_message_id`), что вызывает ошибку `400 Bad Request`. Прошивка формирует JSON вручную и добавляет `message_thread_id` явно.
2. **Перегрев из-за micro-chunking.** Библиотека отправляет JSON мелкими кусками по 2–6 байт, на каждый из которых ESP32 генерирует новый SSL-кадр с шифрованием mbedTLS → процессор уходит в 100%, плата перегревается.

### Реализация (`sendTelegramMessage`)

```cpp
HTTPClient http;
String url = "https://api.telegram.org/bot" + BOT_TOKEN + "/sendMessage";
http.begin(secured_client, url);
http.addHeader("Content-Type", "application/json");
http.setTimeout(10000);

StaticJsonDocument<512> doc;
doc["chat_id"] = CHAT_ID;
doc["text"] = msg;
if (threadId валиден)
    doc["message_thread_id"] = threadId.toInt();

String payload;
serializeJson(doc, payload);
int httpCode = http.POST(payload);   // одним буферизированным запросом
```

**Ключевые свойства:**
- Формирование JSON через `ArduinoJson` (а не конкатенация строк — надёжнее с экранированием).
- Один POST-запрос целиком — никаких микро-чанков.
- Успех определяется кодом `200`.
- Таймаут 10 секунд.
- При `httpCode < 0` (сетевая ошибка) выводится `errorToString`, иначе — тело ответа сервера.

### Предусловия отправки

```cpp
if (!isWifiConnected) return false;
if (!isTimeSynced)   return false;
```

Без синхронизированного времени сертификат `api.telegram.org` будет отклонён — сообщение не уйдёт. Поэтому стартовое сообщение отправляется только **после** успешного NTP.

### Типы сообщений (тексты настраиваются в веб-интерфейсе)

| Назначение | Конфиг-ключ | Значение по умолчанию |
|------------|-------------|------------------------|
| Запуск, сеть есть | `msg_startup_power` | `🚀 ⚡️ ✅ ESS Watcher запущен! Сеть 220В есть.` |
| Запуск, от АКБ | `msg_startup_battery` | `⚡️ ❌ ⚠️ ESS Watcher запущен! Питание от АКБ. ❌` |
| Сеть восстановлена | `msg_power_restored` | `⚡♻️ Сеть 220В ВОССТАНОВЛЕНА!` |
| Отключение сети | `msg_power_lost` | `🚨 ОТКЛЮЧЕНИЕ ПИТАНИЯ 220В!` |

## 11. BuzzerManager — пассивный зуммер и RTTTL

Файл `include/BuzzerManager.h`. Класс-конечный автомат: разбирает RTTTL-строку и **неблокирующе** проигрывает мелодию, продвигаясь по нотам в `loop()`.

### Аппаратная часть

- Пин `GPIO 2` подключается к KY-006 через LEDC-канал 0 (аппаратный ШИМ ESP32).
- `ledcSetup(channel, freq, 8)` — 8-битное разрешение, частота меняется под каждую ноту.
- Громкость регулируется скважностью: `duty = (255 * volume) / 200` — максимум 50% заполнения (оптимально для пьезо).

### Приватные поля

| Поле | Назначение |
|------|------------|
| `currentMelody` | Текущая RTTTL-строка |
| `parsePos` | Текущая позиция парсера |
| `defaultDuration`, `defaultOctave`, `bpm` | Значения по умолчанию из заголовка RTTTL |
| `wholeNote` | Длительность целой ноты в мс: `60000 * 4 / bpm` |
| `isPlaying` | Идёт ли воспроизведение |
| `noteStartTime`, `currentNoteDuration` | Тайминги текущей ноты |
| `noteIsRest`, `restDuration` | Текущая фаза — нота или межнотная пауза |
| `NOTE_FREQS[12]` | Таблица частот 4-й октавы (C–B), далее октавы масштабируются умножением/делением на 2 |

### Приватные методы

| Метод | Назначение |
|-------|------------|
| `playTone(int freq)` | Включить тон заданной частоты (`freq == 0` → тишина). Перенастраивает LEDC и выставляет duty по громкости. |
| `getNoteIndex(char note, bool sharp)` | Преобразует символ ноты в индекс таблицы частот; учитывает диез. |
| `skipWhitespace()` | Пропуск пробелов и запятых в RTTTL. |
| `parseHeader()` | Разбор заголовка `name:d=N,o=N,b=N:`. Устанавливает дефолты и `wholeNote`. |
| `parseAndPlayNextNote()` | Читает **следующую** ноту (длительность, имя, диез, точка, октаву), считает длительность в мс, проигрывает её. Возвращает `false`, если мелодия закончилась. |

### Публичные методы

| Метод | Назначение |
|-------|------------|
| `begin(pin, channel=0)` | Привязка пина к LEDC-каналу, начальный `stop()`. |
| `play(const String& rtttl)` | Запустить RTTTL-мелодию (неблокирующе). Сбрасывает предыдущее состояние, парсит заголовок, играет первую ноту. |
| `beep(freq, duration)` | Одиночный тональный сигнал заданной длительности. Используется редко (м.б. заменён на короткую RTTTL). |
| `stop()` | Остановить воспроизведение, выключить тон, очистить состояние. |
| `playing()` | Возвращает `true`, если мелодия активна. |
| `setVolume(vol)` / `getVolume()` | Громкость 0–100 (с `constrain`). |
| `handleAsync()` | **Главный движок**: вызывается в каждом `loop()`. По `millis()` определяет, пора ли переключать ноту или вставить межнотную паузу. |

### Неблокирующий принцип

Класс **никогда не вызывает `delay()`**. Длительность ноты отсчитывается через `millis()`. Между нотами вставляется короткая «межнотная пауза» длительностью 1/10 от ноты (`playTime = noteDur * 9/10`), что делает мелодию чище (пьезозуммер успевает «затухнуть»).

## 14. Формат RTTTL мелодий

**RTTTL** (Ring Tone Text Transfer Language) — текстовый формат нот, изначально придуманный для рингтонов Nokia. Используется `BuzzerManager` для воспроизведения.

### Структура

```
ИМЯ:d=ЗНАЧЕНИЕ,o=ЗНАЧЕНИЕ,b=ЗНАЧЕНИЕ:НОТА,НОТА,НОТА,...
```

Три части, разделённые двоеточием `:`:
1. **Имя** мелодии (любой текст).
2. **Заголовок с дефолтами:**
   - `d=` — длительность по умолчанию (1=целая, 2=половинная, 4=четвертная, 8=восьмая, 16=шестнадцатая).
   - `o=` — октава по умолчанию (4–7; у нас ограничено).
   - `b=` — темп в BPM (beats per minute).
3. **Ноты** через запятую.

### Синтаксис ноты

```
[Длительность]ИмяНоты[#][.][Октава]
```

| Часть | Описание | Пример |
|-------|----------|--------|
| Длительность (опц.) | Если опущена — берётся из `d=` | `8e` = восьмая ми |
| Имя ноты | `p` (пауза) или `a`–`g` | `c`, `d#`, `p` |
| `#` (опц.) | Диез (повышение на полтона) | `c#` |
| `.` (опц.) | Увеличивает длительность на 50% (пунктирная) | `4c.` |
| Октава (опц.) | Цифра 4–7; если опущена — из `o=` | `c6` |

### Пример

```
Startup:d=16,o=6,b=140:c,e,g,c7,8e7
```
- Имя: `Startup`.
- Дефолты: шестнадцатые ноты, октава 6, темп 140 BPM.
- Ноты: `c`, `e`, `g` (октава 6, длительность 1/16 каждая), `c7` (octave 7), `8e7` (длительность 1/8, октава 7).

### Расчёт длительностей (в `BuzzerManager`)

- `wholeNote = 60000 * 4 / bpm` — целая нота в миллисекундах.
- `noteDur = wholeNote / duration` — длительность текущей ноты.
- Пунктирная нота: `noteDur = noteDur + noteDur / 2`.
- Реально звучит `playTime = noteDur * 9/10`, оставшиеся 10% — межнотная тишина.

| Heartbeat в `loop()` | 5000 мс | Частота отладочного вывода в Serial |

| HTTP-таймаут Telegram | 10000 мс | Защита от зависания запроса |
| Минимальный `delay(10)` | 10 мс | Разгрузка основного цикла |

## 🔗 Краткая шпаргалка по потоку выполнения

```
[Включение питания]
        │
        ▼

   ├─ Монтирование LittleFS

   ├─ Звук "startup"

        │
        ▼
   [Событие GOT_IP] ─▶ NTP configTime()
        │                       │
        │                       ▼
        │              [NTP готов] ─▶ needsStartupMessage = true
        │
        ▼
   loop() (цикл):
   ├─ handleStatusIndicationAsync()   // RGB-пыхи

   └─ Мониторинг питания (каждые 500мс):
```
