## 1. 🟠 `UniversalTelegramBot` — мёртвая зависимость

**Файлы**: `platformio.ini:30`, `src/main.cpp:7`  
**Проблема**: Библиотека `UniversalTelegramBot` (v1.3.0) подключена только ради
единственного макроса `TELEGRAM_CERTIFICATE_ROOT`. Сама отправка сообщений
делается вручную через `HTTPClient.h`. Это лишний код в прошивке и лишняя
зависимость в `platformio.ini`.

**План исправления**:
1. Скопировать PEM-сертификат Telegram в локальную константу (например, в `main.cpp`
   как `static const char TELEGRAM_ROOT_CA[] PROGMEM = R"=(...)=";`).
2. Удалить `#include <UniversalTelegramBot.h>` из `main.cpp`.
3. Удалить `witnessmenow/UniversalTelegramBot @ ^1.3.0` из `lib_deps` в `platformio.ini`.
4. Собрать и проверить, что HTTPS-запросы к Telegram API работают.

**Трудозатраты**: 10 минут.

## 2. 🟠 `playMelody()` всегда падает в fallback — пользовательские RTTTL не используются

**Файл**: `src/main.cpp:75-126`  
**Проблема**: Функция `playMelody(id, powerOn)` строит путь к файлу по схеме
`/sounds/{schema}_{powerOn ? 0 : 1}.txt`:
- id=0 → `standard_0.txt` / `standard_1.txt`
- id=1 → `optimistic_0.txt` / `optimistic_1.txt`
- id=2 → `alarm_0.txt` / `alarm_1.txt`

Но в `/data/sounds/` лежат файлы с совсем другими именами:
`Star-Wars.txt`, `Coca-Cola.txt`, `Crazy-Frog.txt`, `Dallas.txt`,
`Funky-Town.txt`, `Mission-Imposible.txt`, `Pililipopom.txt`,
`Танец-Маленьких-Утят.txt`, `system.txt`.

**Ни одного файла с суффиксами `_0`/`_1`!**
Поэтому `LittleFS.open()` всегда возвращает false, и проигрывается
жёстко закодированная RTTTL-строка из fallback-блока.

**Важно**: `system.txt` читается отдельно через `SoundStorage::getSystemSound()` —
это работает корректно (используется для startup и wifi_mode звуков).

**Дополнительный нюанс**: суффикс `_0` означает "сеть есть" (powerOn=true),
а `_1` — "сети нет" (powerOn=false). Это неочевидная логика именования,
стоит либо документировать, либо инвертировать.

**План исправления**:

### Вариант A: Переименовать существующие файлы (проще)
1. Выбрать 3 мелодии для «включения света» (powerOn=true) и 3 для «выключения» (powerOn=false).
   Например:
   - standard_0.txt ← Coca-Cola.txt (весёлая)
   - standard_1.txt ← Dallas.txt (спокойная)
   - optimistic_0.txt ← Star-Wars.txt
   - optimistic_1.txt ← Funky-Town.txt
   - alarm_0.txt ← Mission-Imposible.txt
   - alarm_1.txt ← Crazy-Frog.txt
2. Переименовать/скопировать файлы в `/data/sounds/`.
3. Проверить: `SoundStorage::listSounds()` должен показать новые имена.

### Вариант B: Переделать схему имён в коде (гибче)
1. Добавить в `SoundStorage` статический метод `getMelodyName(schemaId, powerOn)`,
   который возвращает правильное имя файла из таблицы соответствия.
2. Хранить маппинг в коде как `PROGMEM const char* melodyMap[3][2]`.
3. Изменить `playMelody()` — использовать `SoundStorage::getMelodyName()`.

### Вариант C: Хранить выбор мелодии как имя файла в NVS
1. Добавить в `Config` структуру поле `melody_on_file[32]` и `melody_off_file[32]`.
2. В веб-интерфейсе дать выпадающий список файлов из `/sounds/`.
3. `playMelody()` читает имя файла напрямую из конфига.

**Рекомендация**: Вариант A для быстрого фикса, вариант B для правильной архитектуры.

**Трудозатраты**: 30-60 минут.

## 3. 🟡 `delay(5000)` в `setup()` — жёсткая блокировка на 5 секунд

**Файл**: `src/main.cpp:317`  
**Проблема**: `delay(5000)` в начале `setup()` полностью блокирует
микроконтроллер на 5 секунд. За это время невозможно вывести ничего в Serial,
нет WiFi, нет индикации.

**План исправления**:
1. Убрать `delay(5000)`.
2. Если задержка нужна для стабилизации USB/UART — мигать светодиодом
   в цикле с `millis()` (FastLED инициализируется до этого).
3. Или просто уменьшить до `delay(500)` — достаточно для стабилизации питания.

**Трудозатраты**: 5 минут.

## 4. 🟡 GPIO 1 (Battery Voltage) заявлен, но не реализован

**Файлы**: `README.md:52`, `src/main.cpp`  
**Проблема**: README говорит: «GPIO 1 — Battery Voltage». В коде GPIO 1
не инициализирован и не читается. Функция мониторинга заряда АКБ отсутствует.

**План исправления**:
1. Инициализировать GPIO 1 как `INPUT` (с делителем напряжения).
2. Добавить `analogRead(PIN_BATT_VOLT)` в `loop()` (с периодом, например, 60 сек).
3. Рассчитывать напряжение с учётом коэффициента делителя:
   ```
   V_batt = analogRead(PIN) * 3.3 / 4095 * DIVIDER_RATIO
   ```
4. Добавить порог «низкий заряд» (< 3.4В для Li-Ion 18650).
5. При низком заряде отправлять предупреждение в Telegram.
6. Обновить веб-интерфейс: показывать текущее напряжение батареи.

**Важно**: ESP32-C3 АЦП — 12 бит (0-4095), опорное ~1.1В (требуется аттенюатор).
Требуется калибровка под конкретный делитель.

**Трудозатраты**: 1-2 часа (включая тестирование с реальной батареей).

## 5. 🟡 `beep()` перекрывает текущую RTTTL-мелодию

**Файл**: `include/BuzzerManager.h:227`  
**Проблема**: Метод `beep(freq, duration)` вызывает `stop()`, который сбрасывает
конечный автомат RTTTL, и вручную управляет LEDC через `ledcWriteTone()`.
Если в этот момент играет мелодия — она прерывается без возможности восстановления.

**На практике** : `beep()` вызывается только в `setup()` (до начала `loop()`)
и в `enterConfigMode()`. Вероятность коллизии в текущей архитектуре низкая,
но архитектурно это проблема.

**План исправления**:
1. В `beep()` проверить `isPlaying`. Если да — прервать мелодию вызовом `stop()`,
   затем выполнить `beep()` (текущее поведение, но осознанное).
2. Либо вообще не использовать `beep()` в пользу коротких RTTTL-строк
   (`play("beep:d=4,o=6,b=200:4c")`).

**Рекомендация**: отказаться от `beep()` в пользу `play()` с короткой RTTTL.
Это сохранит архитектуру чистым конечным автоматом.

**Трудозатраты**: 15-20 минут.

## 6. 🟡 Пароль WiFi в NVS открытым текстом

**Файл**: `include/ConfigManager.h:14`  
**Проблема**: `Config.wifi_pass[64]` хранится в NVS как есть. При физическом
доступе к устройству злоумышленник может дампнуть NVS и получить пароль.

**План исправления** (опционально):
1. Добавить XOR-маску с ключом, хранящимся в коде.
2. Применять маску перед `putBytes()` и после `getBytes()`.

**Рекомендация**: принять риск как низкий. Для ESP32-C3 с аппаратным
Flash Encryption это избыточно.

**Трудозатраты**: 5 минут (документировать) или 30 минут (реализовать XOR).

## 7. 🟢 `handleStatusIndicationAsync()` использует static — не потокобезопасно

**Файл**: `src/main.cpp:131`  
**Проблема**: `static unsigned long lastBlinkCycleTime = 0;` внутри функции.
На однопоточном Arduino это безопасно, но если в будущем код портируется
на многозадачную среду (FreeRTOS), возникнут гонки.

**План исправления**:
1. Вынести `lastBlinkCycleTime` в глобальную область рядом с другими
   переменными состояния.
2. Если использовать FreeRTOS — защитить мьютексом или вынести в задачу.

**Трудозатраты**: 2 минуты.

## 8. 🟢 `delay(10)` в конце `loop()` — избыточен

2. Если нужен контроль частоты цикла — использовать `millis()` и
   неблокирующее ожидание.

**Трудозатраты**: 1 минута.

## 10. 🟢 Расхождение README и кода: цвет индикации

**Файлы**: `README.md:26`, `src/main.cpp:142-148`  
**Проблема**: README утверждает:
- «Светодиод загорится **фиолетовым**» в режиме настройки.
- «**Двойной синий** "пых" раз в 1.5 сек при активном подключении.

В реальном коде:
- В режиме настройки (`isConfigMode`) светодиод — **синий**, не фиолетовый.
- Синий цвет используется **только** в config mode, а не при активном WiFi.
- При активном WiFi и наличии 220В — **зелёный**, без сети — **красный**.

**План исправления**:
1. Исправить `handleStatusIndicationAsync()`: добавить фиолетовый цвет
   для config mode (как в README) ИЛИ исправить README на синий (как в коде).
2. Убрать из README строку про «двойной синий пых при WiFi» — такого нет.
3. Либо реализовать синюю индикацию WiFi в коде (что было бы логично).

**Рекомендация**: исправить README в соответствии с кодом (синий для config mode),
а индикацию WiFi оставить как есть (зелёный/красный по статусу питания).

**Трудозатраты**: 2 минуты.

## 11. 🟢 `playMelody()`: неочевидная логика суффиксов `_0` / `_1`

**Файл**: `src/main.cpp:84`  
**Проблема**: Суффикс `_0` означает «сеть 220В есть» (powerOn=true), `_1` — «сети нет»
(powerOn=false). Это обратная интуитивному пониманию логика (обычно `1` = есть,
`0` = нет). Нет комментария, объясняющего это.

**План исправления**:
1. Добавить комментарий в код, поясняющий логику: `_0` = есть сеть, `_1` = нет сети.
2. Или инвертировать: `String suffix = powerOn ? "_1" : "_0";`
   (и переименовать файлы соответственно).

**Рекомендация**: просто задокументировать, т.к. инверсия сломает совместимость
с существующими NVS-настройками мелодий.

**Трудозатраты**: 1 минута.

## 12. 🟢 Утечка real credentials в репозиторий

**Файл**: `include/secrets.h`  
**Проблема**: файл содержит реальные credentials (SSID: EnCRT, пароль WiFi,
токен бота, chat_id, thread_id). Хотя `.gitignore` исключает его из коммитов
и история git чиста, само наличие реальных данных в исходниках — риск утечки
при любом сбое в `.gitignore` или при передаче проекта третьим лицам.

**План исправления**:
1. Заменить credentials в `secrets.h` на заглушки (как в `secrets.example.h`).
2. Хранить реальные credentials отдельно (например, в переменных окружения
   или в защищённом файле вне проекта).
3. Удалить `secrets.h` из git-индекса, если он там был (сейчас не был,
   но профилактически): `git rm --cached include/secrets.h`.

**Трудозатраты**: 5 минут.
