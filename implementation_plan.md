# Архитектурный план и документация ESS Watcher (Pro Edition)

Этот документ описывает внутреннюю логику, инженерные решения и архитектуру проекта. Он служит "идеальным промптом" для понимания кодовой базы и дальнейшего развития системы.

## 1. Аппаратные решения (Схемотехника)

### Пины и периферия ESP32-C3

- **GPIO 0 (Sense):** Пин мониторинга сети 220В. **Важно:** ESP32-C3 работает на логике 3.3В. Модуль UPS выдает 5В. Подключение напрямую сожжет порт! Обязательно использование резистивного делителя напряжения (например, 10кОм и 20кОм), чтобы понизить 5В до безопасных ~3.3В.
- **GPIO 2 (Buzzer):** Управление пассивным пьезозуммером KY-006. Используется аппаратный ШИМ-канал ESP32 (`ledcSetup`/`ledcWriteTone`) для генерации тонов на разных частотах. Воспроизведение мелодий реализовано через RTTTL-парсер (`BuzzerManager`), который неблокирующе проигрывает ноты из строк формата RTTTL. Звуковые файлы хранятся в LittleFS (`/sounds/`) и загружаются через `SoundStorage`.
- **GPIO 10 (WS2812B):** Пин для управления умным RGB-светодиодом (через библиотеку FastLED).
- **GPIO 8 (Встроенный LED):** Изъят из логики. На ESP32-C3 это _strapping pin_, участвующий в процессе загрузки. Изменение его состояния во время бута приводило к сбоям.

### Энергоэффективность и температурный режим

- По умолчанию Wi-Fi модуль ESP32 работает на максимальной мощности (20 dBm), что приводит к скачкам потребления тока и перегреву чипа (до 60+ градусов). В коде мощность принудительно снижается до `WIFI_POWER_8_5dBm` сразу после коннекта.
- Включен режим сна модема `WiFi.setSleep(WIFI_PS_MIN_MODEM)`. Это снижает тепловыделение и экономит заряд 18650 при работе в автономном режиме.

## 2. Программная архитектура (Software Stack)

### Событийно-ориентированный Wi-Fi

Вместо классических "блокирующих" проверок в `loop()`, используется системное прерывание `WiFi.onEvent()`.
Когда система ловит `ARDUINO_EVENT_WIFI_STA_GOT_IP`, она взводит флаги `needsTimeSync` и `isWifiConnected`. Вся тяжелая обработка (NTP и SSL) перенесена в основной цикл `loop()`, чтобы избежать переполнения стека задачи `arduino_events` (ее размер всего ~4KB, а для SSL нужно >6KB).

### Синхронизация времени (NTP)

Телеграм работает исключительно по HTTPS. Проверка SSL-сертификатов сервера `api.telegram.org` невозможна, если часы на чипе показывают 1970 год (сертификат просто отклоняется как "еще не действительный").
Функция `syncTime()` обращается к пулу `pool.ntp.org` (с резервным прямым IP-адресом на случай сбоя DNS). Пока время не синхронизировано, отправка любых сообщений запрещена.

### Telegram API (HTTPClient vs UniversalTelegramBot)

Мы отказались от использования встроенной функции `bot.sendMessage()` для отправки уведомлений по двум критическим причинам:

1.  **Поддержка Топиков (Supergroups):** Стандартная библиотека версии 1.3.0 не поддерживает параметр `message_thread_id`, принимая его за `reply_to_message_id`, что вызывает ошибку Telegram API `400 Bad Request`.
2.  **Глубокий перегрев (Micro-chunking):** Библиотека отправляла собранный JSON мелкими кусками (по 2-6 байт). На каждый кусок ESP32 вынуждена была генерировать новый SSL-кадр с шифрованием mbedTLS. Это вешало процессор в 100% и раскаляло плату.

**Решение:** Мы формируем JSON-документ вручную через `ArduinoJson`, добавляем туда параметр `message_thread_id` и отправляем его одним буферизированным POST-запросом через встроенный `HTTPClient`. Это происходит мгновенно, надежно и поддерживает любые новые фичи API Telegram.

## 3. Онлайн-конфигурация и Хранение

- **ConfigManager (NVS)**: Вместо жесткого кодирования макросов, используется библиотека `Preferences`. При первом запуске данные копируются из `secrets.h` в NVS. Все последующие изменения через веб-интерфейс сохраняются только в NVS.
- **WebPortal (Captive Portal)**: При удержании кнопки настройки (GPIO 9) на 2 секунды, ESP32 переходит в режим `WIFI_AP`. Запускается `DNSServer` для перехвата всех DNS-запросов (Redirect to 192.168.0.11) и `WebServer` для отображения формы настройки.
- **Индикация режимов**:
  - **Зелёный**: Сеть 220В есть.
  - **Красный**: Работа от АКБ.
  - **Синий**: Режим настройки (AP Mode).
  - Индикация реализована в виде двойного «пыха» раз в 2 секунды.

## 4. Безопасность и Конфигурация (Git)

Проект использует локальный файл `include/secrets.h` для хранения SSID, паролей и токенов. Этот файл внесен в `.gitignore` и никогда не попадет в публичный репозиторий GitHub.
Для новых пользователей создан файл-шаблон `include/secrets.example.h`, который содержит пустые макросы и является инструкцией по самостоятельной настройке.
